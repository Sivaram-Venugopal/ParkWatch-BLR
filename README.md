# ParkWatch BLR — AI-Driven Parking Intelligence for Congestion Enforcement

Tata Technologies InnoVent submission — Problem Statement 1: Poor Visibility on
Parking-Induced Congestion.

## What this is

A system that processes 298,450 anonymized illegal-parking violation records
from Bengaluru Traffic Police (Nov 2023 – Apr 2024) and turns them into:

1. A **Congestion Impact Score (CIS)** for every violation — a transparent,
   documented proxy for how much that specific parking event likely
   obstructed traffic, since the raw data has no speed/flow measurements.
2. **2,293 hotspot zones** (150m grid cells) ranked by an **Enforcement
   Priority Index (EPI)** that combines violation volume, persistence across
   days, and severity ceiling.
3. A **Random Forest model** (R² = 0.91) that predicts expected congestion
   impact for any location/time/vehicle-type combination, including ones not
   directly observed in the historical data.
4. An interactive **dashboard** (map, leaderboard, charts) for a reviewer or
   enforcement planner to explore the results with no setup required.

## How to run the pipeline (regenerate everything from raw data)

Requirements: Python 3.10+, ~4GB RAM free.

```bash
pip install pandas numpy scikit-learn joblib
python pipeline.py
```

This reads the CSV from `data/`, runs all seven stages (clean → parse → score
→ zone → rank → train → export), and writes everything into `artifacts/`.
On a single CPU core, the full run takes roughly 8–10 minutes; the Random
Forest training step (Stage 6) is the slowest part.

Note: if you change `n_estimators` in the `RandomForestRegressor` call, more
trees = better accuracy but longer training time. The default (150 trees) was
chosen specifically to balance accuracy against the very constrained
single-core, ~4GB environment, used during development — increase it freely if
running on better hardware.

## How to view the dashboard (no installation needed)

The dashboard is a fully self-contained static site — Leaflet and Chart.js
are vendored locally in `docs/vendor/`, so it does **not** depend on any
CDN for its core libraries (only the map background tiles require an active
internet connection, which is standard for any web map).

**Easiest way — just open the file:**

1. Open `docs/index.html` directly in any modern browser (Chrome,
   Firefox, Edge) by double-clicking it.
2. That's it. No server, no build step, no installation.

**If your browser blocks local file scripts** (rare, but some strict browser
security settings do this), run a simple local server instead:

```bash
cd docs
python3 -m http.server 8000
```

Then open `http://localhost:8000` in your browser.

## What the dashboard shows

- **Map**: all 687 Critical/High priority hotspot zones, sized and colored by
  EPI. Click any marker for zone-level detail (dominant violation type,
  vehicle type, police station, day count).
- **Leaderboard**: searchable, filterable table of all 2,293 qualifying
  zones, ranked by EPI.
- **Temporal chart**: violations by hour of day, with the night/early-morning
  enforcement skew visually flagged in red.
- **Violation and vehicle breakdowns**: citywide composition charts.
- **Station ranking**: total congestion impact by police jurisdiction.
- **Methodology panel**: the three honest caveats about what the data can
  and cannot show (no ground-truth congestion data, night-skewed detection,
  why grid zoning was used instead of density clustering).

## Methodology summary

See `artifacts/metadata.json` for the full machine-readable methodology
(severity weights, vehicle weights, time weights, EPI formula, model
performance). In brief:

- **CIS** = violation severity weight × vehicle footprint weight × time-of-day
  traffic weight. Non-congestion violations (defective plate, tinted glass,
  fare disputes) are excluded entirely — they get CIS = 0 and are filtered
  out before zoning.
- **Hotspot zoning** uses fixed 150m × 150m grid cells, not density-based
  clustering (DBSCAN was tested and rejected after it chained 55% of all
  citywide violations into a single cluster spanning multiple police
  jurisdictions — a known failure mode on unevenly dense street data).
- **EPI** = 0.5 × normalized total impact + 0.3 × normalized persistence
  (violations per active day) + 0.2 × normalized severity ceiling (worst
  single event in the zone).
- **Predictive model**: Random Forest regressor on
  [lat, lon, hour (sin/cos), day-of-week (sin/cos), weekend flag, police
  station, vehicle type, zone id] → CIS. R² = 0.91, MAE = 0.59 on a held-out
  20% test split.

## Known limitations (disclosed, not hidden)

1. No ground-truth traffic speed/flow/volume data exists in the source
   dataset — CIS is a documented proxy, not a measured congestion delay.
2. ~75% of detections fall between 9pm and 8am, which reflects when
   enforcement devices/patrols are active, not necessarily when illegal
   parking peaks. Daytime parking impact during actual commute congestion
   hours is likely undercounted in raw volume (though CIS's time weighting
   partially compensates).
3. 42% of records have no `validation_status` outcome recorded (pending
   review) at the time this dataset was exported; the model uses all
   records regardless of validation status, since restricting to
   "approved" only would have discarded nearly half the data and there is
   no evidence the unvalidated half is systematically different.
